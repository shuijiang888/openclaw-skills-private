# 《Cursor 新手指南：零基础把工作做成》

面向**不懂代码、但想用 Cursor 把工作做成**的用户。可作为操作手册与工作方法论：何时用聊天、何时用 Agent、如何描述任务、如何验收、如何避险。

---

## 1. 这份指南适合你吗

**适合**，如果你：

- 有明确的工作目标（写方案、整理信息、做页面原型、改文档、排查问题、协助部署思路等）
- 愿意**说清楚要什么、不要什么、成功长什么样**
- **不要求**立刻会写代码，但愿意打开文件、复制粘贴、偶尔在终端执行 AI 给出的命令（也可请他人代执行）

**不完全适合**，如果你：

- 希望 100% 零操作、AI 直接登录你所有账号替你完成一切（受安全与平台限制，现实做不到）

---

## 2. Cursor 是什么（业务语言）

Cursor 本质是：**带 AI 的代码编辑器 + 项目工作台**。

- 能读你电脑里的**项目文件夹**（workspace）
- 你能用自然语言说明：**目标、约束、验收标准**
- 能在项目里：**查找、修改、生成文件、运行命令、解释报错**（能做到多少取决于授权与环境）

可理解为：**“会读你资料的项目经理 + 工程师 + 文档助理”**，但需配合正确的工作方式。

---

## 3. 安装与第一次打开

### 3.1 用「项目文件夹」打开

- 尽量用 **Open Folder** 打开目录，例如：`~/Projects/我的项目/`
- 这样 AI 才能稳定“看见”文件结构

### 3.2 固定放置重要说明

在项目根目录建议有：

- `README.md`（项目是干什么的）
- `AGENTS.md` 或 `.cursorrules`（希望 AI 怎么做事、禁区是什么）

不必写得很技术，**写清楚即可**。

---

## 4. 界面：常用的 5 个入口

名称可能随版本略有差异，概念通用。

| 入口 | 用途 |
|------|------|
| **Chat** | 提问、要解释、讨论方案、读某文件 |
| **Composer / Agent** | 直接改项目、多文件任务、更接近“交付” |
| **文件树** | 查看 AI 改了哪些文件 |
| **Terminal** | 安装、启动、看日志；可：**AI 给命令 → 复制运行 → 输出贴回** |
| **Diff** | 看具体改动；变更多时宜抽查关键处 |

---

## 5. 不懂代码也能用好：核心工作法

本质是把任务**工程化**。

### 5.1 每次任务推荐结构

1. **目标**：要达成什么（一句话 + 可验收）
2. **背景**：现状、相关人/系统
3. **约束**：时间、成本、不能做之事（隐私/合规/风格）
4. **输入材料**：文件路径、链接、截图、表格
5. **输出物**：页面？文档？报告？脚本？
6. **验收标准**：怎样算完成（列表）
7. **风险**：最担心什么

### 5.2 成功标准要具体

- 弱：「帮我优化一下」
- 强：「改成可给老板汇报版：不超过 8 节、每节有小标题、结论先行、附风险与下一步」

### 5.3 分步与检查点

可对 AI 说：

- 「先输出计划，不要改文件」
- 「先做最小可用，再扩展」
- 「每完成一步等我确认」

## 6. 高效提问模板（可复制）

### 研究类

```text
目标：我要一份可执行结论，用于【场景】
已知信息：【粘贴或文件路径】
限制：不编造；不确定请标注「需验证」
输出：1)结论 2)依据 3)风险 4)下一步
```

### 写作/整理类

```text
目标：把【材料】整理成【文档类型】
读者：【谁在看】
风格：【正式/口语/销售】
结构：【目录要求】
验收：必须含【要点A/B/C】
```

### 改项目类（不懂代码也可）

```text
目标：在仓库实现【功能/页面/流程】
约束：少改无关文件；变更要有说明
验收：
- 步骤1 → 期望结果1
- 步骤2 → 期望结果2
请先给计划与将影响的文件清单，再动手。
```

---

## 7. 上下文怎么给

- **指路径**优于只粘贴全文（长文档尤甚）
- **少目标**：一次一个主目标，多目标拆会话或分阶段
- **决策写下来**：`CHANGELOG.md`、`DECISIONS.md` 等管理文件亦可由 AI 维护

---

## 8. 规则与记忆

`AGENTS.md` / `.cursorrules` 可写：

- 行业与术语偏好、输出语言、文风
- **隐私红线**（哪些不能进仓库、不能外传）
- 协作节奏（先计划 / 直接做 / 步步确认）

最低成本：**README + 每次大改后更新 CHANGELOG**。

---

## 9. 任务闭环

1. 写清**验收标准**  
2. **执行**（AI 改文件/出方案）  
3. **逐条验证**  
4. **固化**（文档/截图/发布说明）  
5. **复盘**（问题与规避）

---

## 10. Git（最低认知）

- **提交（commit）**：存档点  
- **分支（branch）**：平行时间线，降低搞坏主线风险  

大改前可请 AI 说明「为何要存档」；提交信息可请 AI 起草，由你决定是否提交。

---

## 11. 常见坑

- 混淆「本地能跑」与「已上线」
- 一次改太多、验收不过 → 应用**里程碑**
- **密钥、密码、私钥**勿写入仓库 → 用环境变量或密码管理器
- 终端命令：先问「改什么、能否回滚」；慎对 `rm`、`sudo`、全盘类操作

---

## 12. 安全与隐私红线

- 不向 AI 提供：银行卡、身份证、机密原件、客户隐私表、账号密码、私钥
- 敏感资料先**脱敏**再讨论
- 对外截图前遮住个人信息与内部系统标识

---

## 13. 第一周上手路线

| 天 | 内容 |
|----|------|
| 1 | 用「目标-约束-验收」模板提 3 个任务 |
| 2 | 让 AI 读 `README.md` 并补全项目说明 |
| 3 | 完成纯文档/资料整理任务（熟悉 Diff） |
| 4 | 完成小改动（文案、结构） |
| 5 | 做一次「验证清单」（运行/检查结果） |

---

## 14. Chat 还是 Agent？

| 情况 | 用 |
|------|-----|
| 解释、讨论 | Chat |
| 改文件并交付 | Composer/Agent |
| 跑命令验证 | Agent + Terminal 或复制运行 |
| 长文档 | Chat 出大纲 → Agent 写入文件 |
| 任务很杂 | 先要计划与里程碑 |

---

## 15. 不写代码也能交付的成果类型

- 方案、制度、SOP、培训材料、汇报稿  
- 需求说明、流程图文字版、结构化纪要  
- 原型说明、页面结构与文案  
- 资料整理、对比表、调研结论（含待验证项）  
- 把系统需求拆成执行清单（再交工程师）

---

## 结语

驾驭 Cursor 的关键三件事：

1. **任务说清楚（可验收）**  
2. **上下文给对（文件与约束）**  
3. **坚持闭环（验证 + 固化 + 复盘）**
