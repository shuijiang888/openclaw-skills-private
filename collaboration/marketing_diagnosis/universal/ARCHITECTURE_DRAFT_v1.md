# 通用行业营销诊断系统：可执行架构草案 v1

## 1. 文档目的

本草案用于研发、产品、运维三方对齐：

1. 第一阶段上线范围（独立可运行，先上云）
2. 与现有平台的边界关系（门户 vs 诊断系统）
3. 第二阶段 CRM（纷享销客）对接策略

本文强调「可执行」：只写可落地的最小集合，避免泛方案。

---

## 2. 目标与非目标

### 2.1 阶段目标（P1）

1. 不依赖纷享销客，完成问卷答题、落库、查阅、导出。
2. 兼容三行业页面：
   - 医疗器械：`/diag/h5_medical.html`
   - 能源电力：`/diag/h5_energy.html`
   - 智能制造：`/diag/h5_manufacturing.html`
3. 门户可稳定跳转，发布与回滚流程可复用现有门禁。

### 2.2 非目标（P1 不做）

1. 不做复杂 BI 中台。
2. 不做 CRM 双向同步编排平台。
3. 不做行业动态规则引擎在线编辑器（先用版本化静态配置）。

---

## 3. 与现有平台关系（关键边界）

### 3.1 角色划分

- 现有门户（profit-web）：只负责入口导航与访问管理。
- H5 静态页：负责问卷交互和基础前端体验。
- 诊断后端服务（新增）：负责数据持久化、评分快照、报表快照、查询 API。
- CRM 对接服务（可同服务）：负责异步同步线索/商机。

### 3.2 约束

1. 诊断业务逻辑不得耦合到现有盈利系统业务表。
2. 前端不得直连 CRM；CRM 密钥不得进入前端。
3. 问卷版本与评分版本必须入库，保证历史可追溯。

---

## 4. 建议部署拓扑（腾讯云）

### 4.1 推荐落地

1. 静态页面：COS + CDN（或保留 Nginx 静态目录）
2. API 服务：CVM 容器化 Node.js（Fastify/NestJS/Express 均可）
3. 数据库：MySQL 8（或 PostgreSQL，二选一）
4. 可选组件：Redis（幂等与异步任务）

### 4.2 网络与域名建议

- 门户域名（现有）：`portal.example.com/profit`
- 诊断静态页：`portal.example.com/diag/*`
- 诊断 API：`diag-api.example.com/v1/*`（推荐独立子域）

---

## 5. 第一阶段 MVP 功能范围

## 5.1 用户侧

1. 进入行业 H5，完成问卷。
2. 10 步轻诊断 -> 初估页 -> 解锁完整版续答。
3. 生成报告（得分、等级、维度、TOP3、Q_B2 展示）。
4. 预约专家（二次确认）。
5. 断点续答（`localStorage`）。

### 5.2 后端侧

1. 提交问卷并落库（含版本信息与评分快照）。
2. 查询单份提交结果。
3. 记录线索并通过 `lead_kind` 区分两类语义：
   - `diagnosis_summary`
   - `expert_opportunity`
4. 基础统计接口（按行业、时间）。

### 5.3 数据契约补充

1. 允许 H5 `type + payload` 原样透传，服务端以 `diag_submission.payload_json` 持久化原始请求快照。
2. `diag_lead` 建议同样保留 `payload_json`，便于后续 CRM 映射调试与重放。

---

## 6. 版本治理（必须执行）

每条提交记录必须包含：

- `industryEdition`（行业）
- `questionnaireVersion`（题库版本）
- `scoringModelVersion`（评分模型版本）

若任一版本缺失，提交请求应直接拒绝（400），避免脏数据。

---

## 7. API 设计原则

1. 服务端重新计算分数，不信任前端总分。
2. 接口幂等：支持 `submissionId` 或 `requestId` 幂等键。
3. 所有写接口记录审计日志（来源、UA、IP、时间）。
4. 输出响应包含标准错误码，便于前端稳定提示。

---

## 8. CRM 二期对接（纷享销客）

## 8.1 接入原则

1. 仅服务端调用纷享接口。
2. 同步采用异步任务队列（失败可重试）。
3. 不阻塞用户提交主链路。

### 8.2 最小实施顺序

1. 先同步线索（Lead）
2. 再同步商机（Opportunity）

---

## 9. 安全与合规建议

1. 手机号、姓名按最小必要原则存储。
2. 对外日志不打印完整手机号（脱敏）。
3. 数据库与备份启用加密存储。
4. 设置数据保留策略（例如 12/24 个月归档）。
5. 导出权限限制为内部授权角色。

---

## 10. 发布与回滚建议

### 10.1 发布门禁

1. `/diag` 三页面可访问（200 + 关键文案探针）。
2. API 健康检查通过（DB 连通、迁移状态正常）。
3. 门户卡片全部指向新 URL，不指向历史页面。
4. Nginx include 链条硬校验通过（沿用现有脚本门禁）。

### 10.2 回滚策略

1. 静态文件回滚到上一个版本。
2. API 服务回滚到上一个镜像。
3. 数据库迁移坚持向前兼容，不做破坏性回滚。
