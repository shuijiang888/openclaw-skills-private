# collab-001 复盘（2026-04-06）

## 过程回顾

- OpenClaw/小江在仓库约定 `CURRENT_TASK` + `HANDOFF` + `AGENTS` 的协作路径与安全红线。
- Cursor 按 `HANDOFF` 创建 `collaboration/playbook.md`（六个一级标题），并更新 `AGENTS.md` 对齐「仅文件驱动」与阻断规则。
- `STATUS.md` 从 `in_progress` 推进到 `state: done`，`task_id` 与 HANDOFF 一致。
- `./scripts/check-collab.sh` 作为门禁：先因缺少 `playbook.md` 失败，补齐并通过后提交。

## 遇到的问题

- **脚本失败**：初版验收脚本要求 `playbook.md` 与 `state: done`；在文档未落地前失败属预期，按 HANDOFF 逐项补齐即可恢复通过。
- **文件角色混淆风险**：任务文件与 `STATUS` 模板结构相近，误把 YAML 块写进 `CURRENT_TASK.md` 会降低可读性；应用固定标题区分「任务说明」与「运行状态」。

## 协议有效性

- **有效**：以 `HANDOFF` 为单一执行来源、结果写回 `STATUS`、脚本可重复验收，跑通一轮闭环。
- **待强化**：`check-collab.sh` 与 `task_id` 强绑定，每轮需同步更新脚本或改为按 `CURRENT_TASK`/`HANDOFF` 声明的检查项生成，否则易遗漏。

## 改进建议

- 用 `collaboration/TEMPLATE.md` 固定 OpenClaw 侧写字段，降低漏项与格式漂移。
- `check-collab.sh` 与模板对齐：每轮在 HANDOFF 中明确「本轮脚本检查项」，避免隐式假设仍为上一 `task_id`。
- `CURRENT_TASK.md` 保持纯任务叙述；运行态只放在 `STATUS.md`。

## collab-002 新增发现（追加到上方改进建议）

- `check-collab.sh` 应动态感知本轮任务边界，而非硬编码上一轮检查项
